# AI Architecture Assessment Report

## 1. Current AI Behavior
*   **Invocation**: The system utilizes `CopilotKit` on the frontend, which acts as the orchestrator for LLM interactions. The backend `CopilotkitController` facilitates the bridge via an `OpenAIAdapter` connecting to an internal LLM gateway ("Bifrost").
*   **LLM Tasks**: Currently handles UI manipulation (creating/updating dashboards) via specific actions (`updateDashboardUI`) and performs NL queries against connected datasets.
*   **Integration Flow**:
    *   **Frontend**: `useDashboard.ts` defines actions and readables passed to the Copilot context.
    *   **Backend**: NestJS controller manages the LLM request/response cycle.
    *   **Observability**: `LangfuseService` handles prompt versioning (`dashboard-assistant-instructions`) and LLM request tracing.
*   **Logic Split**: Logic is primarily orchestrational on the backend, while client-side aggregation (`dashboard-utils.ts`) handles the final visualization data transformation.

## 2. AI Failure Modes and Limitations
*   **Brittle Intent Parsing**: The `updateDashboardUI` action forces the LLM to map natural language to ~18 flat scalar configuration parameters. Minor nuances in user intent often lead to configuration mismatches.
*   **Context Pressure**: Large dataset inputs passed via `useCopilotReadable` consume significant context window, leading to potential hallucination or loss of data column awareness.
*   **Lack of Grounding**: There is no explicit validation layer for the JSON output generated by the LLM. It generates `dataJson` and `configSeries` parameters that the frontend trustfully renders; if schemas are malformed or column names change, the UI crashes or displays empty charts.

## 3. Agentic Extension Opportunities
| Agent Type | Problem Solved | Input Need | Output Produce | Architecture Fit |
| :--- | :--- | :--- | :--- | :--- |
| **Web Search Agent** | Answers questions outside the local dataset. | User Query | Search Results / Snippets | Feasible (moderate) |
| **Data Enrichment** | Augments local data with external API info. | Local Data Row Keys | Enriched Data JSON | Feasible (moderate) |
| **Industry Analysis** | Provides benchmarks for current chart data. | Local Trend Data | Benchmarking Insights | Feasible (high) |

## 4. Real-world Knowledge & External Data
*   **Value Add**: The dashboard currently operates in a vacuum. Integrating external market or industry data would allow for comparative analysis (e.g., "how does my KPI compare to industry average?").
*   **Risks**: Trust and citation are major risks. Without a clear ground-truth source for retrieved data, hallucination or misinterpretation of benchmark data could severely impact user trust.

## 5. Data Visualization Intelligence
*   **Current State**: Aggregations (`sum`, `avg`, `min`, `max`, `count`) are handled in `dashboard-utils.ts` on the client side.
*   **Needs**: To support intelligent comparisons, a structured schema registry is required. The AI currently 'guesses' column types based on samples; a explicit registry would ensure the AI understands column semantics (e.g., which columns are temporal vs. nominal).

## 6. Architecture Implications
*   **Orchestration**: Adding agents requires a move towards a multi-step orchestration model. The current flat request-response flow in `CopilotkitController` would need to evolve to support agentic loops.
*   **Storage**: Currently, AI-generated insights are transient. A new storage layer for "AI Insights" or "Agentic Analysis" would be needed if users want to persist multi-step research findings.
*   **Validation Layer**: A mandatory schema validation step between LLM output and frontend state is required to prevent rendering brittle/malformed UI elements.

## 7. Assessment Classification
| Capability | Classification | Reasoning |
| :--- | :--- | :--- |
| **GenUI Actions** | Already supported | Core function of current controller. |
| **Schema-aware context** | Feasible (moderate) | Requires structured metadata registry. |
| **Web Retrieval** | Feasible (moderate) | Can use `useCopilotAction` hook. |
| **Multi-step Agents** | Feasible (high complexity) | Requires persistent state for agent loops. |
| **Hard-coded Parsing** | Not a fit | System is natively LLM-driven; avoid hard-coding. |
