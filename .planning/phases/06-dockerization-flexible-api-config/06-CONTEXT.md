# Phase 6: Dockerization and Flexible API Configuration without Rebuilds - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Containerize the full-stack CopilotKit AI Dashboard (Frontend, Backend, Database) for one-command deployment on any OS. Pre-built images are hosted on GHCR. End users configure all API keys, models, and services via `.env` at runtime — no image rebuilds needed.

</domain>

<decisions>
## Implementation Decisions

### Container Architecture
- **D-01:** Four services: `db` (postgres:15-alpine), `backend` (custom NestJS), `frontend` (custom Next.js), `bifrost` (maximhq/bifrost). Langfuse uses cloud (https://cloud.langfuse.com), not self-hosted.
- **D-02:** Database auto-creates tables via TypeORM `synchronize: true`. No migration step required.
- **D-01a:** Bifrost runs at `http://bifrost:8080/v1` (internal Docker network). Backend's `BIFROST_BASE_URL` points to it.
- **D-01b:** Bifrost data persisted via volume mount (`./bifrost-data:/app/data`) so provider configs survive restarts.

### Image Distribution
- **D-03:** Images pushed to GHCR under `ghcr.io/magggiiii/copilot-dashboard-backend` and `ghcr.io/magggiiii/copilot-dashboard-frontend`.
- **D-04:** Images must be set to **public** visibility so anyone can pull without authentication.
- **D-05:** GitHub repo: `https://github.com/magggiiii/dashboard-demo.git`
- **D-06:** CI/CD via GitHub Actions — auto-build and push on `main` branch pushes and version tags.

### Environment Variable Strategy
- **D-07:** Keep variable names as `BIFROST_*` (not renamed to `AI_*`).
- **D-08:** Required user input: `BIFROST_API_KEY` (their Groq API key) only. Everything else has working defaults.
- **D-08a:** Default model/provider stays the same as our setup: `BIFROST_MODEL=groq/llama-3.3-70b-versatile`.
- **D-08b:** `BIFROST_BASE_URL` defaults to `http://bifrost:8080/v1` (internal Bifrost container). User does NOT need to set this.
- **D-08c:** The setup script must configure the Groq provider inside Bifrost. This means either: (a) a pre-baked `config.json` mounted into the Bifrost container with the user's API key, or (b) using Bifrost's Web UI at `http://localhost:8080` for manual setup.
- **D-09:** `JWT_SECRET` auto-generated by setup script (openssl rand).
- **D-10:** `DB_HOST` overridden to `db` (Docker service name) in docker-compose.yml.
- **D-11:** Langfuse keys are optional. Setup script prompts: "Do you have Langfuse API keys? (y/n)". If yes, user pastes `LANGFUSE_SECRET_KEY` and `LANGFUSE_PUBLIC_KEY`. If no, tracing is disabled gracefully.
- **D-11a:** `LANGFUSE_BASE_URL` defaults to `https://cloud.langfuse.com` (Langfuse Cloud).
- **D-11b:** Setup script instructs users to: create account at cloud.langfuse.com → create project → copy API keys.
- **D-12:** `NEXT_PUBLIC_API_URL` defaults to `http://localhost:3500` — entrypoint script handles runtime replacement.

### Frontend Docker Strategy
- **D-13:** Next.js build uses placeholder `__NEXT_PUBLIC_API_URL_PLACEHOLDER__` baked at build time.
- **D-14:** `docker-entrypoint.sh` performs `sed` replacement of placeholder with runtime `$NEXT_PUBLIC_API_URL` before starting server.
- **D-15:** `output: 'standalone'` must be added to `next.config.ts` for Docker-optimized builds.
- **D-16:** `turbopack.root: '../'` must be removed/guarded for Docker build context compatibility.

### Backend Docker Strategy
- **D-17:** Multi-stage build: install deps + build → copy dist + node_modules → run `node dist/main`.
- **D-18:** `bcrypt` native compilation handled by adding `python3`, `make`, `g++` to build stage.

### Cross-Platform Setup
- **D-19:** `setup.sh` (Mac/Linux) and `setup.ps1` (Windows PowerShell) — interactive terminal scripts.
- **D-20:** Scripts check for Docker installation. If missing, print: "Please install Docker from: https://docs.docker.com/get-started/get-docker/"
- **D-21:** Scripts have two tiers: Required (AI keys) → Advanced (OAuth, Langfuse, AWS).
- **D-22:** Scripts offer to run `docker compose pull && docker compose up -d` at the end.

### Networking
- **D-23:** Backend port 3500 exposed to host (browser makes client-side requests to `NEXT_PUBLIC_API_URL`).
- **D-24:** Database port NOT exposed externally (backend connects internally via `db:5432`).
- **D-25:** Frontend port 3000 exposed to host.

### Convenience Tooling
- **D-26:** Include a `Makefile` with targets: `setup`, `start`, `stop`, `logs`, `restart`, `clean`.
- **D-27:** Include a Docker deployment section in the project README.

### Code Changes Required
- **D-28:** Add `output: 'standalone'` to `gui-dashboard-frontend-main/next.config.ts`.
- **D-29:** Remove/guard `turbopack.root` in `gui-dashboard-frontend-main/next.config.ts`.
- **D-30:** Add S3 credential guard in `gui-dashboard-backend-feature-langfuse/src/modules/s3/s3.service.ts` to prevent crash when AWS is not configured.

### Bifrost Provider Setup
- **D-31:** Bifrost must be pre-configured with Groq as a provider. The setup script will generate a `bifrost-data/config.json` file containing the user's Groq API key and the virtual key mapping.
- **D-32:** The Bifrost Web UI is exposed at `http://localhost:8080` so the user can manage providers and view logs after setup.
- **D-33:** The `BIFROST_API_KEY` in the backend `.env` is the Bifrost **virtual key**, which maps to the real Groq key inside Bifrost's config.

### Langfuse Cloud Setup
- **D-34:** Langfuse is NOT self-hosted. Users create a free account at https://cloud.langfuse.com.
- **D-35:** Setup script guides users to create a project and generate API keys.
- **D-36:** If user skips Langfuse, `langfuse.service.ts` gracefully disables tracing and logs a warning. System works fine without it.

<code_context>
## Existing Code Insights

### Key Patterns
- Backend uses `ConfigService.get()` for all env vars — fully compatible with Docker env injection
- Langfuse gracefully disables itself when credentials are missing (good pattern)
- S3 does NOT gracefully handle missing credentials (needs guard)
- Frontend uses `NEXT_PUBLIC_API_URL` in 7 files — all client-side (browser) requests

### Integration Points
- `docker-compose.yml` → Backend reads `DB_HOST=db` to connect to Postgres container
- `docker-entrypoint.sh` → Replaces placeholder in `.next/static/**/*.js` before server start
- `setup.sh` / `setup.ps1` → Creates `.env` from `.env.example` and runs `docker compose up`

</code_context>
